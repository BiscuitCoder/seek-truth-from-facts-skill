# On Practice — Verification and Action Tool

The central thesis of On Practice: **practice is the source of knowledge, and the sole criterion for testing whether knowledge is correct.** Knowledge is not an endpoint — it is a tool for guiding practice, and it is corrected through practice.

## The Source and Limits of Knowledge

**Knowledge originates from practice**: Without direct experience, knowledge is limited. In analysis, distinguish between:
- Knowledge from direct practice (firsthand experience, real tests, actual data)
- Knowledge from indirect experience (others' experience, books, theoretical models)
- Knowledge from inference (logical extensions of known information)

All three have value, but reliability decreases in that order. Label the quality of the knowledge source when stating conclusions.

**Awareness of limits**:
- Knowledge is always formed under specific conditions; when conditions change, knowledge may no longer apply
- Do not treat knowledge from one stage as eternal truth
- "It worked before" does not mean "it will work now"

## Perceptual and Rational Knowledge

**Perceptual knowledge**: Direct, concrete, surface-level knowledge. Derived from the senses and direct experience; the starting point of cognition.
- Manifests as: phenomena, cases, data points, intuition
- Value: real, concrete, persuasive
- Limit: fragmentary; can be misled by surface appearances

**Rational knowledge**: Concepts, judgments, and laws formed through abstraction and generalization. A deepening of perceptual knowledge.
- Manifests as: laws, models, frameworks, principles
- Value: systematic, predictive
- Limit: may be over-abstracted and detached from specific context

**The correct cognitive path**: Perceptual knowledge → abstraction and generalization → rational knowledge → return to practice for verification

**Common errors**:
- Only perceptual knowledge, unable to extract patterns ("I feel like it should work")
- Only rational knowledge, ignoring concrete reality (applying models while ignoring data)
- Jumping from perceptual knowledge directly to action, skipping rational analysis

## The Spiral Ascent of Practice → Knowledge

Cognition develops not in a straight line but in a spiral ascent:

```
Practice → Perceptual knowledge → Rational knowledge → New practice → Deeper knowledge → ...
```

Each cycle returns to the starting point at a higher level, with knowledge continuously deepening.

**Implications for analysis**:
- Current knowledge is a product of a certain stage — it is not the endpoint
- For complex problems, recommend phased progression rather than planning everything at once
- After each practical phase, re-examine whether the original understanding needs revision

**Designing phase verification checkpoints**:
At the end of each action phase, answer:
1. Did the practical outcome match the original understanding?
2. Which judgments were confirmed, and which were falsified?
3. Based on new practice, how should the understanding and actions of the next phase be adjusted?

## Practice as the Criterion of Truth

The correctness of analysis and judgment is ultimately tested by practice — not by logical consistency or authoritative endorsement.

**Implications for analysis**:
- When giving action recommendations, build in a verification mechanism (how will you know this step was right?)
- Judgments that cannot be tested by practice have limited value
- Guard against two extremes:
  - **Dogmatism**: treating a conclusion as absolute truth, beyond question
  - **Empiricism**: trusting only one's own experience, refusing theoretical guidance

---

## Connecting Theory to Reality

The value of theory lies in guiding practice. Theory detached from reality is hollow; practice without theoretical guidance is blind.

**In analysis**:
- When using any framework or theory, explain why it is applicable to the current specific situation
- When theory conflicts with actual data or cases, trust the actual evidence
- When making recommendations, account for the actor's specific resource constraints — do not prescribe idealized solutions detached from reality

## Action Path Design Guide

After feasibility is confirmed, design the action path based on On Practice:

### Principles

1. **Start with the principal contradiction**: The first step must address the core problem, not peripheral details
2. **Phased progression with verification checkpoints**: Each phase has a clear definition of success
3. **Small-scale first, then expand**: Where possible, validate understanding through limited practice before scaling
4. **Build in an adjustment mechanism**: Plans are not fixed; explicitly state under what conditions the overall path should be re-evaluated

### Path Structure Template

```
Phase 1: Core action to resolve the principal contradiction
  - What specifically to do
  - Resources / time required
  - Checkpoint: how to determine this step succeeded
  - Risk: what is the main risk, and how to respond

Phase 2: Proceed after Phase 1 is validated
  - (same structure)

Adjustment triggers:
  - If situation X occurs, re-evaluate the overall path
```

### Prohibited Behaviors

- Investing significant resources in secondary problems before the principal contradiction is resolved
- Setting unverifiable objectives ("improve user experience" without stating how to measure it)
- Replacing concrete actions with subjective effort descriptions ("try hard", "do our best")
- Action paths with no contingency for failure
